# 回答生成过程状态展示 — 产品需求

> **状态**：待开发  
> **适用范围**：智能投研对话页 — 用户提问后、正式回答流式输出前的过程状态展示  
> **关联文档**：`docs/agent/langgraph-flow.md`（后台节点）、`docs/agent/llm-system-prompts.md`（编排层）

---

## 1. 背景与目标

### 1.1 背景

当前用户在等待 Agent 生成回答时，长时间只看到 loading 或空白，无法感知系统正在做什么，容易产生「卡住」的不安感，也难以建立「有据可依」的信任感。

### 1.2 目标

在正式回答开始流式输出之前，以 **纵向时间线** 的形式，向用户逐步展示 Agent 的执行阶段（理解问题 → 识别意图 → 匹配专家 → 获取资料 → 审核 → 生成回答）。

### 1.3 非目标

- 不展示模型原始 CoT、token 数、思考时长等技术细节。
- 不将后台 LangGraph 节点 ID 直接暴露给用户。
- 过程状态展示不是最终回答的一部分，不参与引用与溯源。

---

## 2. 核心交互原则

### 2.1 过程可见，结论优先

执行阶段文案用于 **等待期** 的信息填充；一旦正式回答开始流式输出，这些文案在视觉上应 **让位给回答正文**。

### 2.2 收起而非销毁（关键）

当 response 开始流式输出时：

| 要求 | 说明 |
|------|------|
| **收起隐藏** | 过程状态区域默认折叠，不再占据主视觉区域 |
| **不可销毁** | 已展示过的步骤数据保留，不从页面/会话中删除 |
| **可回看** | 提供角标、摘要条或类似轻量入口，用户点击后可展开查看完整过程时间线 |
| **默认折叠** | 流式输出期间及输出完成后，过程区域均保持折叠态，除非用户主动展开 |

**动机**：流式输出开始后，过程文案与正文同时展开会造成信息冗余、分散注意力；但用户若关心「它读了什么」，仍应能一键回看。

### 2.3 诚实展示

- 仅展示实际发生的步骤与子项，不编造未执行的资料获取动作。
- 澄清、兜底走独立分支，不假装走完主链路。

## 3. 展示时机与生命周期

```text
用户发送问题
    │
    ▼
┌─────────────────────────────────────┐
│  过程状态区（展开，主视觉）           │
│  ● 正在理解您的问题                  │
│  ● 正在识别问题类型                  │
│  ● 正在匹配投研专家 · 问股分析        │
│  ● 正在获取相关资料                  │
│      ✓ 正在查询泸州老窖财务数据       │
│      ✓ 正在检索相关研报              │
│  ● 正在审核回答质量                  │
│  ● 正在生成回答                      │
└─────────────────────────────────────┘
    │
    │  response 首 token 到达，开始流式输出
    ▼
┌─────────────────────────────────────┐
│  [▸ 执行过程]  ← 折叠入口（角标/摘要）  │
├─────────────────────────────────────┤
│  （流式回答正文，主视觉）              │
│  泸州老窖基本面…                      │
└─────────────────────────────────────┘
    │
    │  用户点击「执行过程」
    ▼
┌─────────────────────────────────────┐
│  [▾ 执行过程]                        │
│  ● 正在理解您的问题          ✓       │
│  ● …（完整时间线，只读）              │
├─────────────────────────────────────┤
│  （流式/已完成回答正文）              │
└─────────────────────────────────────┘
```

### 3.1 状态机

| 阶段 | 过程区展示 | 回答区展示 |
|------|-----------|-----------|
| `processing` | 展开，逐步追加步骤 | 空或占位 |
| `streaming` | **自动折叠**，显示回看入口 | 流式输出正文（主视觉） |
| `completed` | 保持折叠，可展开回看 | 完整回答 |
| `clarification` | 展开至澄清步骤后停止 | 展示澄清追问（非主回答流） |
| `fallback` | 展开至兜底步骤后停止 | 展示兜底说明 |

### 3.2 折叠触发条件

- **触发时机**：收到 response 流式首包（或后端 `response_assembly` 开始输出事件）时，立即折叠过程区。
- **动画**：建议轻量过渡（高度收起 / 折叠动画），避免突兀跳变。
- **回看入口**：折叠后保留一行摘要，例如 `▸ 执行过程 · 6 步` 或 `▸ 已匹配问股分析 · 获取 3 项资料`（文案可再定）。

## 4. 主链路步骤与文案

### 4.1 步骤清单

| 顺序 | 步骤 ID | 用户可见主文案 | 文案类型 | 动态规则 |
|------|---------|----------------|----------|----------|
| 1 | `understand_query` | 正在理解您的问题 | 固定 | — |
| 2 | `recognize_intent` | 正在识别问题类型 | 固定 | — |
| 3 | `match_expert` | 正在匹配投研专家 | **动态** | 后缀 ` · {专家类型}` |
| 4 | `fetch_materials` | 正在获取相关资料 | **动态** | 主文案固定；可展开 0～N 条子项 |
| 5 | `quality_review` | 正在审核回答质量 | 固定 | — |
| 6 | `generate_answer` | 正在生成回答 | 固定 | — |

### 4.2 第 3 步：专家类型映射

| 后台 `route_target` | `{专家类型}` |
|---------------------|--------------|
| `stock_analysis_agent` | 问股分析 |
| `data_query_agent` | 数据查询 |
| `hotspot_agent` | 热点解读 |
| `document_qa_agent` | 文档问答 |

展示格式：`正在匹配投研专家 · {专家类型}`

### 4.3 第 4 步：资料获取子项

主文案固定；根据实际 `tool_call` / `rag_retrieval` 展示子项。

| 子项 ID | 文案模板 | 触发条件 | 动态占位符 |
|---------|----------|----------|------------|
| `fetch_market_data` | 正在查询行情与排行数据 | 行情/排行/热力类 tool | — |
| `fetch_financial_data` | 正在查询{标的}财务数据 | 个股财务 tool | `{标的}` |
| `fetch_sector_data` | 正在查询{板块}板块数据 | 板块/概念 tool | `{板块}` |
| `fetch_report` | 正在检索相关研报 | RAG 研报 | — |
| `fetch_announcement` | 正在检索相关公告 | RAG 公告 | — |
| `fetch_news` | 正在检索行业资讯 | RAG 资讯 | — |
| `fetch_document` | 正在检索文档内容 | 文档问答链路 | `{文档}`（可选） |

**子项规则**：

- 先结构化数据（tool），后非结构化资料（RAG）。
- 同类型可合并，单次建议最多 3～5 条。
- 子项完成时标记 ✓；无子项时仅展示主文案。

## 5. 分支链路

### 5.1 澄清分支

**触发**：`clarification_check` → `need_clarification = true`  
**行为**：主链路在第 2 步后中断，不展示步骤 3～6。

| 步骤 ID | 主文案 | 动态后缀 `{澄清点}` |
|---------|--------|---------------------|
| `clarify_info` | 正在确认关键信息 | 见下表 |

| 澄清原因 | `{澄清点}` |
|----------|------------|
| 意图不明确 | 请补充您想了解的方向 |
| 缺少股票名称/代码 | 请补充股票名称或代码 |
| 缺少时间范围 | 请补充时间范围 |
| 缺少指标/维度 | 请补充关注的指标或分析维度 |
| 槽位歧义 | 请确认您指的是哪只股票 |
| 缺少文档 | 请指定要解读的文档或研报 |
| 通用 | 需要更多信息才能继续 |

澄清场景下，过程区在展示澄清追问后保持展开（或随追问一并展示）；用户补充信息后重新走主链路。

### 5.2 兜底分支

**触发**：路由至 `fallback_response`  
**行为**：不走步骤 3～6。

| 步骤 ID | 主文案 | 动态后缀 `{原因类型}` |
|---------|--------|----------------------|
| `prepare_fallback` | 正在准备说明 | 见下表 |

| 兜底原因 | `{原因类型}` |
|----------|--------------|
| 证据不足 | 相关资料不足 |
| 合规风险 | 涉及合规限制 |
| 预测类请求 | 无法提供预测性结论 |
| 工具失败 | 数据暂时不可用 |
| 非投研/闲聊 | 超出投研范围 |
| 通用 | 暂无法给出可靠结论 |

兜底说明开始输出时，过程区同样适用 **收起 + 可回看** 规则。

## 6. 步骤展示状态

| 状态 | 含义 | UI |
|------|------|-----|
| `pending` | 未开始 | 不展示 |
| `running` | 进行中 | 灰色节点 + 主文案 |
| `completed` | 已完成 | 绿色节点 + ✓ |
| `active` | 当前步骤 | 高亮/强调 |
| `skipped` | 未触发 | 不展示 |

## 7. 与后台的对接要求

### 7.1 事件来源

过程状态应由后端编排事件驱动（SSE / WebSocket），建议与 Trace `steps[].node` 对齐，但 **前端展示文案经映射层转换**，不直接展示节点 ID。

| 后台节点（参考） | 前端步骤 |
|------------------|----------|
| `context_preprocess` | `understand_query` |
| `intent_recognition` + `slot_extraction` | `recognize_intent` |
| `routing_decision` | `match_expert` |
| `tool_call` / `rag_retrieval` | `fetch_materials`（含子项） |
| `quality_check` | `quality_review` |
| `response_assembly` 开始前 | `generate_answer` |
| `response_assembly` 首 token | 触发折叠 + 开始流式正文 |
| `clarification_response` | `clarify_info` |
| `fallback_response` | `prepare_fallback` |

### 7.2 前端需接收的最小字段

```typescript
// 示意，非最终实现
type ProgressStep = {
  step_id: string;
  status: 'running' | 'completed';
  label: string;           // 已渲染的主文案（含动态后缀）
  children?: ProgressStep[]; // fetch_materials 子项
};

type ProgressEvent =
  | { type: 'step_start'; step: ProgressStep }
  | { type: 'step_complete'; step_id: string }
  | { type: 'response_stream_start' }  // 触发折叠
  | { type: 'response_delta'; content: string };
```

### 7.3 折叠与流式的时序约束

- `response_stream_start` 到达时：过程区 **立即** 折叠，不因「生成回答」步骤未完成而延迟。
- 若 `generate_answer` 步骤尚未标记完成，折叠后回看时间线内该步可显示为进行中或已完成（以实现为准，回看态只读）。

## 8. 折叠入口 UI 要求

| 项 | 要求 |
|----|------|
| 位置 | 紧贴回答正文上方，或消息气泡顶部内侧 |
| 形态 | 单行可点击区域：`▸ 执行过程` + 可选摘要（步数 / 专家类型 / 资料数） |
| 交互 | 点击切换展开/折叠；展开时不遮挡回答滚动 |
| 态 | 流式中与完成后均可用；默认可折叠 |
| 无障碍 | 支持键盘聚焦与 `aria-expanded` |

**摘要示例**（任选或组合）：

- `▸ 执行过程 · 6 步`
- `▸ 问股分析 · 已获取 3 项资料`

## 9. 验收标准

### 9.1 主流程

- [ ] 用户提问后，按顺序展示 6 步主链路文案；第 3、4 步含动态内容。
- [ ] 第 4 步子项仅在实际调用 tool/RAG 时出现，且完成后有 ✓。
- [ ] response 开始流式输出时，过程区自动折叠，正文成为主视觉。
- [ ] 折叠后可通过入口展开，完整回看时间线（只读）。
- [ ] 流式输出完成后，过程区仍默认折叠。

### 9.2 分支

- [ ] 澄清场景：展示「正在确认关键信息 · …」，不展示后续主链路步骤。
- [ ] 兜底场景：展示「正在准备说明 · …」，不假装获取资料。
- [ ] 兜底/澄清输出开始时，同样触发折叠 + 可回看。

### 9.3 体验

- [ ] 过程区折叠/展开有过渡，无布局跳动。
- [ ] 未执行的步骤与子项不出现。
- [ ] 同一会话内切换历史消息时，每条消息独立保留其过程时间线状态。

## 10. 示例

### 10.1 问股（流式前 → 流式后）

**流式前（展开）**

```text
● 正在理解您的问题
● 正在识别问题类型
● 正在匹配投研专家 · 问股分析
● 正在获取相关资料
    ✓ 正在查询泸州老窖财务数据
    ✓ 正在检索相关研报
    ✓ 正在检索相关公告
● 正在审核回答质量
● 正在生成回答
```

**流式后（折叠）**

```text
▸ 执行过程 · 问股分析 · 3 项资料
────────────────────────────────
泸州老窖基本面整体…（流式正文）
```

### 10.2 澄清

```text
● 正在理解您的问题
● 正在识别问题类型
● 正在确认关键信息 · 请补充股票名称或代码

（展示澄清追问，无后续主链路步骤）
```

### 10.3 兜底

```text
● 正在理解您的问题
● 正在识别问题类型
● 正在准备说明 · 无法提供预测性结论

（展示兜底说明；输出开始时折叠）
```
